# Расширение прототипа артефактами BPMN

## Контекст
На текущем этапе прототип ориентирован на основной control flow:
- tasks
- events
- gateways
- sequence flows
- lanes

Следующий потенциальный класс расширений:
- `DataStoreReference`
- `TextAnnotation`
- связанные с ними `Association` и `DataAssociation`

## Главный вывод
Расширение в эту сторону возможно без ломки общей архитектуры, но при одном важном условии:
основной процесс должен быть отделён от preview/layout модели через `Process IR`.

Это означает:
- семантика артефактов должна жить в IR
- layout артефактов должен быть отдельным шагом
- экспорт в BPMN должен строиться из IR и layout result, а не из состояния React Flow

## Оценка сложности
### Что добавляется сравнительно легко
- новые сущности в `Process IR`
- новые renderer shapes в React Flow
- новые edge semantics для связей с артефактами
- сериализация semantic BPMN элементов

### Что сложнее
- автолейаут вторичных артефактов
- читаемое расположение аннотаций и datastores вокруг основного потока
- генерация BPMN DI waypoints и bounds так, чтобы схема оставалась понятной

## DataStoreReference
### Семантика
`DataStoreReference` должен представлять внешнее хранилище или устойчивый источник данных, связанный с шагами процесса.

### Как хранить в модели
Рекомендуется хранить как отдельный artifact / node-подобную сущность в `Process IR`, но не как элемент основного control flow.

### Как показывать на графе
Рекомендуемый принцип:
- не включать `DataStoreReference` в основной поток dagre на равных правах с task/gateway/event
- располагать как secondary artifact рядом с целевым шагом
- использовать отдельные связи типа `dataAssociation`

Практический визуальный вариант:
- datastore рисуется сбоку от task
- чаще справа, сверху или снизу от основного узла
- при необходимости допускается отдельная техническая зона в пределах lane

### Как будет выглядеть в BPMN
Поддерживается стандартно:
- semantic element `dataStoreReference`
- связи через `dataAssociation`, если они поддерживаются на целевом уровне export
- отдельные BPMN DI bounds и waypoints

## TextAnnotation
### Семантика
`TextAnnotation` должна использоваться для поясняющего текста, который LLM считает полезным добавить к конкретному элементу процесса.

### Как хранить в модели
Рекомендуется хранить как отдельный artifact с обязательной привязкой к target element.

LLM должна определять:
- нужен ли annotation
- к какому элементу он относится
- какой текст должен быть внутри annotation

LLM не должна определять координаты annotation.

### Как показывать на графе
Рекомендуемый принцип:
- annotation не должна участвовать в расчёте основного потока
- annotation размещается вторым проходом после раскладки основного control flow
- annotation соединяется тонкой `association` line с целевым элементом

Практический визуальный вариант:
- annotation располагается справа или сверху от target element
- не должна ломать глобальные колонки основного потока
- не должна участвовать в rank/column logic dagre

### Как будет выглядеть в BPMN
Поддерживается стандартно:
- semantic element `textAnnotation`
- semantic `association`
- отдельные BPMN DI bounds и waypoints

## Почему главный риск не в XML
И `DataStoreReference`, и `TextAnnotation` являются штатными BPMN сущностями.

Поэтому главный инженерный риск не в semantic export, а в визуальной части:
- как располагать их на графе
- как не ломать читаемость диаграммы
- как не превращать схему в перегруженную сетку вспомогательных объектов

## Рекомендуемая стратегия реализации
### Этап 1. Семантическое расширение IR
Добавить в `Process IR`:
- `dataStoreReference`
- `textAnnotation`
- `association`
- `dataAssociation`

### Этап 2. Рендеринг secondary artifacts
В UI:
- добавить новые shape types
- рисовать их отдельно от main flow
- раскладывать вторым проходом после основного dagre layout

### Этап 3. BPMN export
Поддержать:
- semantic BPMN elements
- DI bounds для artifacts
- DI waypoints для associations

### Этап 4. Улучшение layout
После базовой поддержки улучшать:
- правила размещения datastore
- правила размещения text annotation
- борьбу с пересечениями и визуальным шумом

## Требование к LLM
LLM должна возвращать только семантические привязки:
- нужен ли datastore
- к какому шагу он относится
- нужен ли annotation
- к какому элементу он относится
- какой текст должен быть показан

LLM не должна управлять геометрией diagram layout.

## Итоговая рекомендация
Расширение прототипа `DataStoreReference` и `TextAnnotation` выглядит реалистичным и архитектурно естественным.

Но делать это стоит только после введения `Process IR`, потому что:
- эти сущности относятся к доменной семантике, а не к preview-only модели
- для них нужен отдельный artifact layout
- BPMN export должен использовать чистую внутреннюю модель, а не состояние canvas
